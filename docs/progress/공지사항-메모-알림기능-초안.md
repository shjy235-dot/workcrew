# 공지사항/메모 등록 시 알림 기능 (초안)

## 계획
- 목표: 관리자가 공지사항을 등록/수정하거나, 특정 작업에 현장 메모가 새로 등록되면
  작업자(worker)들에게 알림이 가도록 한다. (이번 작업은 **설계 초안까지만**, 구현 안 함)
- 상태: 초안 검토 중 (구현 보류)

## 1. 알림이 필요한 이벤트
| 이벤트 | 발생 위치 | 알림 대상 |
|---|---|---|
| 공지사항 등록/수정 | `Dashboard.jsx` → `project_info.notice` | 해당 프로젝트에 접속 중인 작업자 전체 |
| 현장 메모 신규 등록 | `DayModule.jsx` → `tasks.memo` (한 줄 추가) | 해당 프로젝트 작업자 전체 (관리자 제외) |

- 공지사항은 "수정"도 잦을 수 있으므로, 타이핑 중 매 keystroke마다 알림이 나가면 스팸이 됨
  → 기존 IME 리얼타임 디바운스 패턴([[realtime-ime-fix]] 참고)처럼 **입력 멈추고 일정 시간(예: 2~3초) 후 최종 값 기준으로만 알림 트리거** 필요.
- 메모는 "등록" 버튼 클릭 시점(현재도 `handleMemoSubmit`으로 확정 시점이 명확함)에 알림 트리거하면 되므로 디바운스 이슈 없음.

## 2. 기술 방식 후보 비교

### 방식 A. Web Push (브라우저 푸시 알림) — 추천
- 앱이 꺼져 있어도(백그라운드/브라우저 종료 후에도) 기기에 알림 도착
- 필요 요소:
  1. Service Worker 등록 (`public/sw.js`) — 현재 프로젝트는 PWA 미설정이므로 신규 추가 필요
  2. VAPID 키 쌍 생성 (`web-push` 라이브러리) → 공개키는 프론트, 비밀키는 Vercel 환경변수
  3. 작업자가 최초 접속 시 "알림 받기" 권한 요청 → 구독 정보(`PushSubscription`)를 Supabase
     신규 테이블 `push_subscriptions` (worker_name, project_id, endpoint, p256dh, auth)에 저장
  4. 공지사항/메모 변경 시 `api/index.js`(Express, 이미 존재하는 백엔드)에
     `POST /api/notify` 같은 엔드포인트 추가 → 해당 프로젝트 구독자 목록 조회 후
     `web-push` 라이브러리로 발송
  5. 트리거 방식은 둘 중 택1:
     - (5-a) 프론트에서 저장 성공 직후 직접 `/api/notify` 호출 (구현 단순, 추천)
     - (5-b) Supabase DB Webhook/트리거로 서버리스 함수 자동 호출 (프론트 코드와 분리되어 더 견고하지만 설정 복잡도 ↑)
- 제약사항:
  - iOS Safari는 **홈 화면에 추가된 PWA**에서만 푸시 지원 (iOS 16.4+), 일반 사파리 탭에서는 불가
    → 작업자에게 "홈 화면에 추가해주세요" 안내가 사실상 필수
  - Android/데스크톱 Chrome은 별도 설치 없이도 지원

### 방식 B. 인앱 알림만 (앱을 열어둔 상태에서만 뱃지/토스트)
- Supabase Realtime을 이미 쓰고 있으므로 구현이 훨씬 간단 (신규 인프라 불필요)
- 단, **앱을 보고 있지 않으면 알림을 놓침** → "알람"이라는 요구사항 취지와는 다소 거리 있음
- 최소 구현: TopBar에 종 아이콘 + 안읽음 뱃지, Realtime으로 notice/memo 변경 구독 후 뱃지 갱신

### 방식 C. 카카오톡/문자(SMS) 알림
- 확실하게 도달하지만 카카오 알림톡/문자 API는 유료 + 사업자 인증 등 절차 필요
- 소규모 현장 작업 앱 규모에는 과함 (비용 대비 효과 낮음)

## 3. 추천안
- **1차: 방식 B(인앱 알림)로 빠르게 먼저 검증** → 별도 인프라 없이 Realtime만으로 구현 가능,
  현재 앱을 "작업 중 계속 켜두는" 사용 패턴(현장 체크리스트 앱)과도 잘 맞음
- **2차: 방식 A(Web Push)로 확장** → 작업자가 앱을 안 보고 있을 때도 알림이 필요하다고
  판단되면, PWA manifest + service worker + VAPID 세팅을 추가로 진행

## 4. 데이터/코드 영향 범위 (구현 시 예상)
- 신규 테이블: `push_subscriptions` (Web Push 채택 시)
- `projectStore.js`: `notice` 업데이트, `memo` 추가 액션 지점에 알림 트리거 훅 추가
- `TopBar.jsx`: 알림 권한 요청 UI / 종 아이콘 뱃지 (방식 B)
- `api/index.js`: `/api/notify` 엔드포인트 (방식 A)
- `public/sw.js`, `manifest.json`: PWA 최소 설정 (방식 A)

## 미결정 사항 (구현 전 확인 필요)
- 관리자도 알림을 받아야 하는지, 아니면 작업자만 대상인지
- 프로젝트 여러 개(멀티프로젝트) 중 "현재 활성 프로젝트"에 한해서만 알림 보낼지, 아니면 이전에
  참여했던 프로젝트 알림도 계속 받을지

## 5. 방식 채택: B (인앱 알림) — 구현 난이도 평가
- **결론: 가볍고 쉽게 구현 가능.** 새 테이블/새 API 엔드포인트 불필요. `TopBar.jsx` +
  `projectStore.js` 두 파일만 건드리면 됨.
- **재활용 가능한 기존 인프라**: `projectStore.js`의 catch-all realtime 구독
  (`public:any_changes`, event `'*'`, `fetchFromDB` 트리거)이 이미 모든 테이블 변경을
  감지하고 있음 → 이 콜백에 "값 비교" 로직만 얹으면 됨.
- **구현 방향**:
  1. `fetchFromDB()` 호출 직전, 로컬에 이미 갖고 있는 `notice`/`memo` 값과 새로 받아온
     값을 비교
  2. 달라졌으면 zustand에 `hasUnreadNotice` / `hasUnreadMemo` boolean만 세팅 (세분화된
     읽음 추적 X — task별/줄 단위 추적은 과설계, 이 앱 규모엔 boolean 2개로 충분)
  3. TopBar에 종 아이콘 + 빨간 점, 클릭 시 boolean 리셋 + localStorage에 확인 시각 저장
- **주의할 리스크 (안 지키면 꼬임)**:
  - 자기 자신이 수정한 내용에 뱃지가 뜨는 문제 방지: 반드시 **"로컬 상태와 비교 먼저 →
    그 다음에 덮어쓰기"** 순서를 지켜야 함 (낙관적 업데이트가 이미 로컬에 반영되어 있으므로
    순서만 지키면 자동으로 자기 수정은 걸러짐)
  - 작업자 본인이 쓴 메모도 같은 원리로 자동 필터링됨
  - 기존 2초 IME 디바운스 타이머와는 무관하게 독립적으로 동작 (서로 안 건드림)
- 다음 단계(구현 시): 위 순서만 지키면서 `hasUnreadNotice`/`hasUnreadMemo` 2개 boolean +
  TopBar 종 아이콘 UI 추가.
